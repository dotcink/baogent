# AGENTS.md

本文件对整个仓库生效；若更深层目录存在 `AGENTS.md` 或 `AGENTS.override.md`，以后者为准。

## 项目目标

逐步构建一个高度自主的 Agent：模型是内层智能，外围系统负责提供能力与约束，也就是 Harness。

## 分层边界

- `src/model`：仅放模型与 provider 适配层；不要放 agent loop、工具编排或 CLI 逻辑。
- `src/agent`：负责 agent 执行环、工具调用与 bash 能力；不要依赖 `src/entry`。
- `src/entry`：只放 CLI、配置加载与模块装配；保持薄，不要堆放核心逻辑。
- `config/`：配置样例与不同环境配置。
- `examples/`：独立示例与实验；不要反向依赖 `src/entry`。

## 核心规则

- 优先选择能完整解决问题的最简单方案。
- 改动保持小而直接，只解决当前请求，避免顺手重构无关代码。
- 优先复用现有 provider、tool、loop 抽象；不要随意新增临时协议或重复封装。
- 保持清晰分层与单向依赖。
- 不引入职责模糊的命名，如 `util`、`helper`、`manager`，除非其职责在仓库上下文中已非常明确。
- 没有明确复用价值时，不要提前提炼抽象。

## 命名与风格

- 新增类型、函数、模块的命名必须直接反映职责。
- 代码与文档风格优先保持与仓库现有写法一致。
- 表达保持精炼，避免重复解释、重复逻辑和冗余结构。

## 文档与配置同步

- 涉及 CLI 用法、配置格式、默认文件名、环境变量的变更，必须在同一改动中同步更新代码、帮助文本、注释、文档和示例。
- 新增或修改配置项时，必须同步更新 `config/example.toml` 与相关示例配置。

## 校验要求

- 做代码修改后，默认运行与改动范围最匹配的校验。
- 如果没有更窄的检查手段，至少运行 `bun run typecheck`。
- 常用命令：
  - `bun run cli`
  - `bun run example:review-refine`
  - `bun run typecheck`

## 完成标准

- 改动必须与请求直接对应。
- 能补测试或校验时要补；不能补时，要说明原因和剩余风险。
- 完成前自行复查 diff，检查明显回归、分层破坏、命名漂移和重复实现。

## 提交信息

- 使用小写约定式前缀，如 `feat:`、`fix:`、`refactor:`、`docs:`。
- 描述保持简短、直接，优先使用英文。

## 规则沉淀

- 只有长期有效、可跨任务复用、且已证明有价值的要求，才写入仓库规则。
- 发现候选规则时，先提出建议，再决定是否写入 `AGENTS.md` 或对应 skill。
