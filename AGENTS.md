
## 本项目在做什么？
一步步，从 0 开始，从内而外，实现一个高度自主的 Agent。
这里一个理念是，模型是最内层的智能，这里要做的是，一步一步给模型添加能力，套上缰绳，也就是 **Harness**。

## 仓库结构
1. `src/model`：模型与 provider 适配层，负责统一模型调用接口，不承载 agent loop 或 CLI 逻辑
2. `src/agent`：agent 执行环、工具调用与 bash 能力，负责把模型能力编排成可运行的 agent
3. `src/entry`：程序入口、CLI 与配置加载，只做装配与参数接线，不堆放核心业务逻辑
4. `config/`：配置样例与不同环境配置
5. `examples/`：独立示例与实验性用法，不反向依赖 `src/entry`

## 设计原则
1. 简洁：接口最小化，不暴露不必要的方法和成员；代码简洁，不冗余
2. 模块化：重视模块划分，模块内高内聚，模块间低耦合；要有接口化思维，想清楚每个模块的接口是什么，模块之间的依赖关系和通信方式是什么
3. 分层架构：清晰的分层及单向依赖逻辑

## 开发约束
1. 文档与实现一致：CLI 用法、配置格式、默认文件名、环境变量说明变更时，必须同步更新注释、帮助文本和示例
2. 命名与风格一致：新增类型、函数、模块命名要直接反映职责，避免模糊命名；代码和文档风格优先与仓库现有写法保持一致
3. 保持分层边界：`src/model` 不依赖 `src/agent` 或 `src/entry`；`src/agent` 不依赖 `src/entry`
4. 入口层保持薄：不要把核心逻辑堆进 `src/entry`，入口层只负责参数解析、配置加载和模块装配
5. 优先复用已有抽象：已有 provider、tool、loop 边界可以承载需求时，不新增临时协议或重复封装
6. 改配置时同步样例：新增或修改配置项时，必须同步更新 `config/example.toml` 与相关示例配置

## 运行与校验
1. 启动 CLI：`bun run cli`
2. 运行示例：`bun run example:review-refine`
3. 类型检查：`bun run typecheck`
4. 做代码修改后，默认至少运行与改动范围匹配的检查；如无更窄检查手段，至少运行 `bun run typecheck`

## 完成标准
1. 改动必须和请求直接对应，避免顺手重构无关代码
2. 能补测试或校验时要补；不能补时，要明确说明原因与剩余风险
3. 涉及 CLI、配置、默认值、文件名、环境变量的变更，必须同步更新文档、帮助文本、示例和配置样例
4. 提交前应自行复查 diff，检查明显回归、分层破坏、命名漂移和重复实现

## 禁止事项
1. 不为了局部方便打破既有分层或把跨层依赖做成双向依赖
2. 不引入职责不清的 util、helper、manager 一类模糊命名，除非它的职责在仓库上下文中已经非常明确
3. 不在没有复用价值时提炼抽象；先保证职责清晰，再考虑抽象复用

## Git Commit
1. 本项目使用约定式前缀提交，commit message 采用小写前缀加冒号的形式，例如 `feat: ...`
2. 优先使用简短、直接的英文描述，不写冗余说明

## 规则沉淀
1. 后续对话中出现的新要求，只有在长期有效、可跨任务复用、且已经体现出实际价值时，才沉淀为项目规则
2. 发现候选规则时，先提出建议，再确认是否写入 `AGENTS.md` 或对应 skill
